# ScholarNova 本次优化报告

日期：2026-07-02

## 1. 优化目标

- 在资源和限流约束下提升 Asta 定向子集 F1。
- 保持比赛私有配置与公开 BYOK 项目严格分离。
- 完善公开仓库的部署、视觉展示、许可、贡献和安全文档。
- 形成可直接用于三分钟产品视频的过程稿和分镜稿。
- 修复会影响公开 CI、SenseNova 异常处理和测试可信度的问题。

## 2. 比赛检索优化

### 结果

| 指标 | 上一版 | 本次 |
| --- | ---: | ---: |
| Precision | 0.259434 | **0.352313** |
| Recall | **0.367893** | 0.331104 |
| F1 | 0.304288 | **0.341379** |
| Recall@20 | 0.160535 | **0.163880** |
| Recall@50 | **0.204013** | 0.200669 |
| Recall@100 | **0.274247** | 0.254181 |
| 平均 API 调用 | **3.667** | 7.333 |
| 平均延时 | **5.904 秒** | 15.139 秒 |

评测范围为 Asta 官方验证集 18 条确定性子集，不是完整比赛总分。本次 F1 相对上一版提升 12.19%，但 API 调用和延时变高；后续优化应优先改善引文交集排序和调用复用，而不是继续增加调用次数。

### 代码改动

- 修复 MS² 等 BibTeX 风格论文别名未展开的问题。
- 为高基数引文查询加入结果预算，减少无关候选对 Precision 的影响。
- 增加论文别名和结构化结果预算回归测试。
- 保留全局 Semantic Scholar 限流、缓存、重试与熔断。

## 3. Token 计量说明

当前基准的 Token 为 0 是真实结果：18 条查询使用确定性规则和引文图，不需要调用 LLM。产品中的模型辅助查询与论文分析继续记录供应商返回的 prompt、completion、total token 和请求次数。

这符合比赛的效率目标：能用确定性算法解决时不消耗模型 Token，需要语义推理时才调用模型。

## 5. 工程修复

- CI 的后端测试改为离线单元/契约测试，显式排除真实外部 API 集成测试。
- 修正过时的论文与证据 API 测试，使测试对应当前路由契约。
- 前端补充 ESLint 配置，CI Node.js 统一到 20。
- Python 静态检查先执行语法和未定义名称等关键规则，存量样式问题后续渐进清理。
- 修复 LLM 网关图像响应异常分支缺少 `json` 导入的问题。
- 为 SQLAlchemy 前向关系补充类型检查导入，消除关键未定义名称。
- 修复后台检索在数据库初始化失败时引用未赋值 `SearchRun` 的二次异常；失败会被记录且不再破坏已接受请求的响应。

## 7. 已知边界

- 当前 F1 尚未超过文献中 SPAR 的 0.38 参考值，且两者评测设置不同，不能直接作为同榜成绩比较。
- v3 用更高 Precision 换取部分 Recall，并增加了 API 调用和延时。
- 27 个外部 API 集成测试需要网络和相应限额，适合手动执行，不应放入每次公开 CI。
- 前端仍有 7 个 React Hook 依赖 warning，不影响测试与构建，但应在后续重构中逐项消除。
- JCR 与中科院分区只有在接入合法授权数据后才能显示 verified；当前不会推断或伪造。

## 8. 建议的下一轮

1. 对引文图结果增加评测快照日期约束，避免新发表论文污染历史金标准。
2. 对多种结果预算做独立验证集调参，避免只优化当前 18 条子集。
3. 将 API 缓存命中率和限流等待时间纳入比赛效率报表。
4. 扩展官方验证样本后再判断是否接近或超过 0.38。
5. 逐步清理 Ruff 存量样式问题与 React Hook warning。
